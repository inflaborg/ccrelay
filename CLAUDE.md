# Formatting and Linting Guidelines

Before committing code to the repository, you must always run the following commands:

* npm run format
* npm run lint

These commands ensure that the codebase remains consistently formatted and free of problematic code constructs.

If the linter reports any warnings or errors, you must fix them in the source code before creating a git commit. The code must be clean with no remaining linter errors.

# Release Workflow

## Version bump (pre-release)

Starts a new version line. Bump ALL package versions in the monorepo together:

1. Update version in these files:
   - `package.json` (root)
   - `packages/core/package.json`
   - `packages/vscode/package.json`
   - `packages/desktop/package.json`
   - `package-lock.json` (root version + all sub-package entries under `packages`)
2. Add a new section in `CHANGELOG.md` under `[Unreleased]`:
   ```
   ## [X.Y.Z] - YYYY-MM-DD (pre-release)

   Pre-release line for X.Y.Z.
   ```
3. Commit message: `chore: X.Y.Z pre-release version bump`
4. Create PR targeting `main` from `feat/dev`.

## Dev pre-releases (per-PR CI builds)

These are temporary and auto-generated by CI. Tag format: `dev-X.Y.Z-<PR#>`.

## Official release

1. Finalize `CHANGELOG.md` — fill in the full changelog entries and remove `(pre-release)`.
2. Create GitHub Release (`vX.Y.Z`) with all PRs listed and `Full Changelog` diff link.
3. CI builds and attaches VSIX + desktop installers as release assets.
